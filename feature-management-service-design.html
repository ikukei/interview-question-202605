1. Purpose

This document describes a practical design for a Feature Management Service for an e-commerce platform. The system manages feature flags across web portals, backend services, and mobile clients. It needs to support low-latency evaluation, controlled rollout, clear ownership, and explainable decisions.

Because this is a take-home interview project, the design intentionally separates what a production-grade system should support, what the demo implementation should build, and what should be documented but not implemented in the demo.

The goal of the demo is not to rebuild LaunchDarkly. The goal is to show strong engineering judgment: clear architecture, useful APIs, deterministic evaluation behavior, reasonable caching, test coverage, and a path from demo to production.

2. Requirement Summary

  • Thousands of feature flags across more than 100 applications and services
  • Web, backend, and mobile clients
  • High-throughput and low-latency flag evaluation
  • Cost-effective caching as the feature catalog grows
  • A client SDK model that is easy to integrate
  • Management, evaluation, and supporting APIs
  • Observability and debugging support
  • Explainability for each decision, including who, where, which rule, which release, and which snapshot version

3. Design Philosophy

The most important design choice is to split the system into two planes. The control plane manages authoring, validation, publishing, audit, and governance. The data plane serves flag evaluation. It should be fast, highly available, cache-friendly, and independent from the control plane during normal runtime.

For the demo, this split can be represented inside a single Spring Boot application. The code should still keep the boundaries clear through packages and service responsibilities.

4. High-Level Architecture

flowchart LR A["Admin UI / API Client"] --> B["Management API"] B --> C["Metadata Database"] B --> D["Publisher"] D --> E["Versioned Snapshot Store"] E --> F["Evaluation API"] F --> G["In-Memory Snapshot Cache"] H["Java SDK Example"] --> F I["Frontend SDK Example"] --> F F --> J["Evaluation Explain Response"]

Production architecture would extend this with a streaming distribution gateway, edge caches, multi-region deployment, Redis, object storage, audit pipelines, and OpenTelemetry.

The demo should keep the architecture simple: one Spring Boot backend, one local relational database, one in-memory snapshot cache, one Vue admin frontend, one Java SDK example, one frontend SDK example, and focused unit and integration tests.

6. Recommended Demo Scope

  1. Spring Boot backend
  2. Local database using H2 or SQLite
  3. REST APIs for flag management, publishing, evaluation, and explanation
  4. In-memory snapshot cache keyed by environment + appKey + version
  5. Basic Vue frontend for managing and testing flags
  6. Java SDK example
  7. Frontend SDK example
  8. Java unit tests for the evaluation engine
  9. A short README explaining production extensions

Recommended non-goals for the demo include multi-region deployment, real-time streaming distribution, Redis, Oracle-specific stored procedures, full RBAC, multi-language SDK matrix, full observability stack, and large-scale segment import.

8. Database Design

8.1 Production Database

Production should use Oracle. It is suitable for strong consistency, auditability, transactional publishing, and integration with enterprise data governance. The domain layer should avoid Oracle-specific logic so that persistence details remain isolated behind repository interfaces.

8.2 Demo Database

For the demo, use H2 or SQLite. H2 is the recommended choice for a Spring Boot and JPA based project because it needs no external database and works well with tests.

9. Snapshot and Cache Design

The runtime evaluation path should not query normalized flag and rule tables on every request. Publishing creates an immutable snapshot, and the backend keeps the latest snapshot in memory.

cache key = environment + ":" + appKey
cache value = latest ConfigSnapshot

Production can extend this with SDK-side local cache, Redis, object storage, delta sync, and streaming version notifications.

10. Evaluation Model

Evaluation loads the latest snapshot by environment + appKey, finds the requested flag, applies enabled state, evaluates rules by priority, applies deterministic rollout, and returns either the matched variation or the default value.

bucket = hash(flagKey + ":" + subjectKey) % 100
enabled if bucket < rolloutPercentage

This guarantees stable rollout behavior for the same user and flag.

11. Explainability Design

Explainability should be part of the evaluation contract. The demo should implement POST /api/v1/evaluations:explain and return the final value, reason code, matched rule, matched conditions, rollout bucket, release key, snapshot version, and evaluation time.

For production, evaluation traces should be sampled by default. Full traces should be enabled on demand for selected flags, users, or trace IDs.

13. Vue Frontend Design

A Vue frontend is worth including, but it should stay small. Recommended pages are an application selector, flag list, flag edit form, rule editor, publish action, evaluation playground, and explain result panel.

The most valuable screen is the evaluation playground. It lets a reviewer choose an app, environment, and flag, edit JSON context, run evaluation, and see the final value and explanation.

14. Java SDK Example

The Java SDK example should show how a backend service would integrate with the feature management service. For the demo, it can call the remote evaluation API directly.

FeatureClient client = FeatureClient.builder()
    .baseUrl("http://localhost:8080")
    .appKey("checkout-service")
    .environment("local")
    .build();

boolean enabled = client.boolVariation(
    "new-checkout",
    FeatureContext.builder()
        .subjectKey("user-123")
        .attribute("region", "us-east")
        .attribute("platform", "ios")
        .build(),
    false
);

Production SDK behavior would add local snapshot cache, background refresh, streaming updates, offline fallback, and exposure batching.

15. Frontend SDK Example

The frontend SDK example can be a TypeScript utility used by the Vue app or a separate small example. For the demo, remote evaluation is acceptable.

const client = createFeatureClient({
  baseUrl: "http://localhost:8080",
  appKey: "checkout-web",
  environment: "local"
});

const enabled = await client.boolVariation("new-checkout", {
  subjectKey: "user-123",
  attributes: {
    region: "us-east",
    platform: "web"
  }
}, false);

Production frontend SDK behavior would require public client keys, limited scoped snapshots, server-side evaluation for sensitive flags, context minimization, and exposure event batching.

16. Testing Strategy

Java unit tests should be included. They are one of the best ways to prove the core system is correct.

Test TypeScope
Unit testsEvaluation engine, rollout hashing, rule matching
Repository testsOptional, if using JPA
Controller testsBasic API contract
End-to-end demo testOptional, create flag, publish, evaluate

Minimum recommended cases include flag not found, disabled flag, rule match, rule priority, deterministic rollout, explain fields, and snapshot version increments.

17. Observability Design

For the demo, use structured logs, include important fields like appKey, environment, flagKey, snapshotVersion, reasonCode, and traceId, and expose a Spring Boot health endpoint.

For production, add OpenTelemetry tracing, Prometheus metrics, Grafana dashboards, alerting, and sampled evaluation traces.

18. Security and Governance

For the demo, security can be simple: optionally use a static API key and document that production requires stronger controls.

For production, use RBAC by team, app, and environment, approval workflow for production publishes, audit logs, signed SDK credentials, PII minimization, and encryption in transit and at rest.

21. Final Recommendation

For this interview project, the best balance is to build a Spring Boot backend, use H2 locally and document Oracle for production, implement the evaluation engine seriously, add in-memory snapshot caching, include Java unit tests, build a compact Vue frontend, and add both Java and frontend SDK examples.

This scope is large enough to demonstrate architectural thinking and engineering ability, but small enough to complete at home without turning the assignment into an unrealistic platform rebuild.
